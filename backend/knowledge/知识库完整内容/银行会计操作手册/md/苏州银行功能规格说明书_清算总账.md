# 1.3.1 功能概述

1. 科目管理：包括科目维护（入口浏览、查询、EXECL导出）、科目使用权维护、科目调整、转核算处理、科目核算产品信息查询。根据大总账发布的科目变更指令完成科目新增、修改、停用、重开的变更处理。  
2. 总账管理：包括总账历史余额查询、总账历史交易查询、总账准实时余额查询、总账加工、定期处理、账务平衡检查、总账报表。  
3. 内部户管理：内部户是用于核算各级机构内部经济活动的账户，包括内部户控制表维护、内部户维护、手工记账交易、内部户交易明细查询、内部户挂销账查询。  
4. 年终结算：苏州银行 XX 银行年终结算处理由大总账系统负责，核心的年终结算主要处理只是将损益类科目自动结转到某个指定科目，该结转传票不需发送给大总账

# 系统。

5. 总账参数管理：用于设置总账相关的业务参数，包括账套参数、核算机构参数、各应用的挂账科目参数。

# 1.3.2 与核心其他子系统的关系

1. 业务平台：负责提供外币折算汇率、柜员交易操作权限管理、会计引擎组件、机构辖属关系和区域编号信息。现金模块业务涉及的内部户其合法性及余额更新、挂销账账务处理需要会计信息子系统负责。现金模块的登记簿信息日终提供给会计信息子系统进行账卡核对。费用管理模块的费用明细日终提供给会计信息子系统进行总分核对。  
2. 存款子系统：存款业务涉及的内部户其合法性及余额更新、挂销账账务处理需要会计信息子系统负责。日终存款子系统提供存款分户余额资料进行总分核对检查。  
3. 存放同业子系统：存放同业业务涉及的内部户其合法性及余额更新、挂销账账务处理需要会计信息子系统负责。日终存放同业子系统提供存款分户余额资料进行总分核对检查。  
4. 票据子系统：票据业务涉及的内部户其合法性及余额更新、挂销账账务处理需要会计信息子系统负责。

# 1.3.3 与外围系统的关系

# 1. 大总账系统：

1) 苏州银行 XX 银行科目由大总账系统统一维护，然后向各个应用系统发布科目信息。核心的科目仅是全科目表的子集，只保留核心相关业务核算使用的科目。科目的新增、修改、停用、重开都由大总账发起，核心接受大总账指令进行相应处理。

2) 每日日终，核心将当日传票按照九个维度（机构段、责任中心段、会计科目段、参考段、产品段、条线段、内部往来段、来源段、备用段）统计借贷方发生额，发送给大总账系统。

2. 国结系统：核心负责国结系统的对公外汇买卖业务、外币汇款业务核算涉及的内部户的开立和维护。国结系统的对公外汇买卖业务、外币汇款业务交易发生生成会计传票，发送给核心记账。核心对这些会计传票分录进行借贷平衡检查，日终发送给大总账系统。核心提供传票分录记账成功对账清单，供国结系统自行下载进行对账。  
3. 中间业务平台等要求在核心成账的外围系统：核心负责业务涉及的内部户的开立和维护。通过核心会计引擎拆分会计分录，会计信息子系统负责对其交易会计分录进行总账加工。核心提供交易流水记账成功对账清单，供外围系统自行下载进行对账。  
4. 柜面：负责提供用户操作的界面。

# 流程补充说明：

# 1、科目管理：

1) 苏州银行 XX 银行科目维护由大总账系统统一维护，然后向各个应用系统发布科目信息。核心的科目仅是全科目表的子集，只保留核心相关业务核算使用的科目。科目的新增、修改、停用、重开都由大总账发起，核心接受大总账指令进行相应处理。  
2) 对于单纯会计核算引起科目平移、合并引起的科目调整，系统自动调整会计引擎科目组参数、与科目相关的总账参数和相应的内部户记录处理，并生成传票调整相应的总账信息。  
3) 对于涉及外部户产品码变更或客户核算信息变更引擎的科目调整或科目拆分通过转核算处理，根据转核算申请的外部分户信息，生成传票调整相应的总账信息。

# 2、内部户管理：

1) 内部户控制表需要预先设置，业务发生根据传票信息，对于涉及内部户的分录需要检查内部户的合法性并更新内部户信息(如果内部户设置了“联机更新余额”标识，则系统自动更新内部户的余额，否则批量的时候统一更新)。如果内部户不存在，则根据内部户控制表的设置，自动生成内部户。  
2) 如果内部户设置了挂销账属性，业务发生时，则进行挂销账处理；  
3) 系统提供手工记账方式，还提供手工记账模板进行手工记账、或者 EXECL 方式进行手工记账。

# 3、总账管理：

1）核心的各应用子系统（存款、存放同业、现金等），以及要求在核心成账的外围系统（如中间业务）通过会计引擎生成分录传票信息（包括联机和批量），系统自动做传票检查，包括传票要素的合法性检查、单笔交易整套传票借贷平衡检查、单机构传票借贷平衡检查。  
2) 在外围成账的外围系统（如国结系统）则将拆分好的分录传票信息送到核心；

国结系统不遵循交易核算分离原则，即核算出现异常如科目状态异常或内部户异常，交易拒绝，返回失败信息给外围系统。

# 1.5 基本定义

# 1.5.1 科目

# 1、定义：

科目号由 4-8 位数字型字符组成，对应于科目层级有一级、二级、三级科目号，每层科目号对应不同的长度，分别为：一级科目 4 位，二级科目后 6 位，三级科目为 8 位。

# 2、编码规则：

XXXX-XX-XX

一级科目 (4) + 二级科目 (2) + 三级科目 (2)。

# 3、类型及长度：

字符型数字，8位长度。

# 4、举例：

1001 现金（一级）

100103 自助设备占款（二级）

10010301 在行式ATM占款（三级）

10010302 离行式ATM占款（三级）

# 1.5.2 内部户账号

# 1、定义：

内部户是指银行根据本身业务经营管理的需要设置的银行内部专用账户。

2、编码规则：

核算机构（9位）+币别（3位）+科目号（8位）+序号（6位）。

3、类型及长度：

字符型数字，26位长度。

4、举例：

010000000 156 10010302 000001

# 1.5.3 挂账编号

1、定义：

业务需要对部分内部户进行明细管理，故使用挂销账机制对内部户下每一项目进行分类，挂账编号就是此分类，此项目分类下的每笔记账即为挂账明细，挂账编号是一套挂账的唯一标识。统一由核心系统进行编码处理。

2、编码规则：

9位机构号+6位年月日+3位序号

3、类型及长度：

字符型数字，18位长度。

4、举例：

711111110 160330001

# 1.5.4 挂账序号

1、定义：

挂账序号是为了序时地反映每一笔挂账明细而设置的顺序号码。

2、编码规则：

NNNN，在一个挂账编号下顺序编排

3、类型及长度：

数字型，4位长度。

4、举例：

0001-第一笔挂账明细

# 1.5.5 销账序号

1、定义：

销账序号是为了序时地反映每一笔销账明细而设置的顺序号码。

2、编码规则：

NNNN，每一笔挂账编号下顺序编排

3、类型及长度：

数字型，4位长度。

4、举例：

0001-第一笔销账明细

# 1.5.6税率代码

1、定义：

用于定义银行税率种类，目前主要指银行增值税的税率代码。

2、编码规则：

CCC -NNN。

前缀 (3) 编号 (3)

3、类型和长度：

字符型，6位长。

4、举例：

VAT001 - 增值税税率 1，税率为  $0\%$ 。

VAT002-增值税税率2，税率为  $6\%$

VAT003 - 增值税税率3，税率为  $11\%$ 。

VAT004 - 增值税税率 4，税率为  $17\%$ 。

科目的状态变更流程如下所示：（SOD：日始 Start Of Day；EOD：日终 End Of Day）

# 1、科目停用：

1) 在科目的失效日期日始，系统自动对“正常”状态的科目进行停用处理。  
2) 如果科目的分户标识为“Y-内部户”：

a) 对该科目下的内部户控制表进行停用处理。  
b) 检查该科目下的内部户余额情况并进行相应处理。如果内部户余额为0，则对内部户进行注销，如果内部户余额不为0，则置为停用状态。  
c) 如果科目下所有内部户都已注销，则科目置为注销状态。如果科目下仍有内部户未注销，则科目置为停用状态。  
d) 每日批次检查停用状态的内部户如果余额为 0，则置为注销状态。如果科目下所有内部户都已注销，则科目置为注销状态。

3) 如果科目的分户标识为 “N-外部部户” 或 “G-科目”:

a) 如果科目相应的总账余额不为 0 , 则将科目置为停用状态。  
b) 如果科目相应的总账余额都为 0 , 则将科目置为注销状态。

# 2、科目重开的处理原则：

1）已停用或注销的科目可以激活，恢复为正常状态。  
2) 如果科目的失效日期已过期，则置为无限期。  
3) 科目激活，不自动同步激活相应的内部户控制表和内部户相关记录；由手工根据业务需要激活内部户控制表和内部户。

# 3、科目冻结处理规则：（苏州银行 XX 银行的“科目冻结功能”暂封）

1) 科目冻结，仅对用于内部户核算的 3 级科目（分户标识为“Y”）进行冻结。外部户的冻结通过相关的应用子系统处理。

a) 科目冻结，对该科目下的“正常”状态下的内部户控制表进行冻结处理。  
b) 科目冻结，对该科目下“正常“状态的内部户记录进行冻结处理。  
c) 科目解冻，不同步相应的内部户控制表和内部户记录，通过手工对相关的内部户控制表进行解冻操作，手工逐个对相关的内部户记录进行解冻处理。

2) 科目冻结后不允许再发生任何对该科目的记账交易，直至科目解冻为止。

# 1.6.1.3科目调整规则

科目调整包括科目平移、科目合并、科目拆分，由于外部分户转产品或客户核算信息变更引起的科目调整和科目拆分。

由于单纯会计核算的科目平移、科目合并引起的科目调整，不涉及外部户产品变更或客户核算属性变更的，系统提供一键式自动处理。将会计引擎中的核算科目、内部户中的科目以及与业务相关的科目参数（各应用的挂账科目、账套科目参数）由原科目更新为新科目；科目的总账余额通过生成的会计传票信息进行调整。科目调整引起内部户账号改变需要修改未销账的挂销账相关主表的内部户信息，并注明原因（原内部号，调整日期等）。

科目拆分是将原有的一个科目拆分为多个科目（一拆多）。按照科目类型的不同，科

目拆分包括内部户科目拆分和外部户科目拆分、纯科目拆分。对于内部户和纯科目的科目拆分，通过手工记账功能将要拆分的内部户和纯科目余额调整到目标科目的内部户和纯科目中，并产生相应的账务记录。会计引擎中的核算科目、业务相关的科目参数，手工通过相关维护功能将原科目更新为拆入科目。

如果需要纯科目转内部户，通过科目调整，提交科目调整申请。要求新旧科目号相同且新科目顺序号有输入，要求必须授权。

由于外部分户产品变更或客户核算属性变更引起的外部户科目调整或科目拆分，涉及相关外部户的本金、表内计提利息、利息等账务调整，由业务提交转核算处理申请，生成分录传票调整总账余额，并同步更新相关的外部户产品信息和客户核算信息。涉及的会计引擎参数和科目参数需手工预先配置。

核心的科目调整和转核算处理完成，线下通知大总账系统。

# 1. 科目平移

1) 科目平移是指将原科目的余额（包括总账和分户）全部迁移到一个新科目中（一对一）。  
2) 科目平移必须在最底层科目之间进行，且移出科目和移入科目的科目类别、记账方式、分户标识、透支约定属性应一致。  
3) 科目平移后，旧科目设置为停用状态，同时要将科目表、会计引擎中的核算科目、内部户中的科目号、科目的总账余额以及与业务相关的科目参数由原科目更新为新科目。  
4) 科目平移, 可通过提供的 “科目调整” 功能, 自动完成会计引擎中的核算科目、内部户中的科目以及与业务相关的科目参数由原科目更新为新科目; 并且通过内部户余额（内部户科目）或科目总账余额（分户标识为“N”/“G”)生成会计传票凭证调整相应的科目总账余额，调整传票转出方红字反映。科

目调整引起内部户账号改变，需要修改未销账的挂销账相关主表的内部户信息，并注明原因（原内部号，调整日期等）。

5) 纯科目转内部户，可通过提供的“科目调整”功能，提交科目调整申请。要求新旧科目号相同且新科目顺序号有输入，要求必须授权。

# 2. 科目合并

1) 科目合并是将几个科目合并到一个科目中（多合一），并入的科目可以是已存在的科目，也可以新设的科目。  
2) 科目合并实质上是多个科目平移到同一个科目中去，其处理原则与科目平移相同。  
3) 科目合并, 可通过提供的 “科目调整” 功能, 自动完成会计引擎中的核算科目、内部户中的科目以及与业务相关的科目参数由原科目更新为新科目, 以及修改挂销账相关的内部户信息; 并且通过内部户余额 (内部户科目) 或科目总账余额（分户标识为 “N”/“G”) 生成会计传票凭证调整相应的科目总账余额, 调整传票转出方红字反映。科目调整引起内部户账号改变, 需要修改未销账的挂销账相关主表的内部户信息, 并注明原因 (原内部号, 调整日期等)。

# 3. 内部户科目/纯科目拆分：

1) 首先要明确哪些内部户或纯科目（分户标识为“Y”/“G”）需拆分到其他科目内部户或纯科目中，然后通过手工记账功能将要拆分的内部户/纯科目的科目余额调整到目标科目的内部户/纯科目中，并产生相应的账务记录。  
2) 会计引擎中的核算科目、业务相关的科目参数，手工通过相关维护功能将原科目更新为拆入科目，调整传票转出方红字反映。  
3) 如果需要纯科目转内部户，通过科目调整，提交科目调整申请。要求新旧科

目号相同且新科目顺序号有输入，要求必须授权。但是必须首先建立内部户控制表记录，如果科目分户标识为“G”，则要求授权；（该内部户控制记录不允许设置挂销账，如果要求挂销账，待内部户建立好后，通过挂销账属性修改实现），

调整日日终，科目调整账务批量处理

a) 修改科目相关的参数：会计模型-科目组参数、总账参数；  
b) 将科目总账余额转移到内部户，生成传票；  
c) 将科目的分户标识由“G”改为“Y”；

# 4. 外部分户产品或客户核算属性变更引起的科目调整或科目拆分

由于会计核算、统计报表报送等原因引起的科目调整，有可能涉及相关的外部分户产品码信息变更、客户核算属性变更等，涉及相关外部户的本金、表内计提利息、利息等账务调整，由“转核算处理”来完成。对于客户核算属性、存款由于差错造成外部分户的产品码信息变更或客户的核算信息变更，也可由“转核算处理”统一调整。

“转核算处理” 处理流程：

1) 日间联机提供客户层和账户层转核算申请，处理方式有2种：

a) 客户核算属性：联机输入如下信息，客户号、涉及客户的新核算属性（客户类型、金融机构类别、账户性质、资金性质、期限、贷款经营类型、小微标识），转核算申请新增、修改输入画面，增加旧产品码、旧核算属性、旧科目组的返显。在晚上批量的时候，根据客户号取得该客户下的所有账户信息，将该客户下所有账号的相应核算属性进行变更，并进行总账的账务调整；  
b) 账户核算属性：联机输入客户号、账号、涉及产品的新核算属性（客户类型、金融机构类别、账户性质、资金性质、期限、贷款经营类型、小微标识），转核算申请新增、修改输入画面，增加旧产品码、旧核算属性、旧科目组的返显。在晚上批量的时候，根据新的核算属性信息进行相应的转核算处理，

并进行总账账务的调整。

c) 支持联机时以批量上传的形式将多个客户核算属性、账户核算属性批量上传至核心系统，在晚上批量的时候，根据日间上传的客户属性或账户属性进行核算属性变更同时进行总账的账务调整。

2) 参数中心需设置好相关的新科目、会计引擎参数;  
3) 对转核算信息进行合法性检查：

a) 个人产品不允许与单位/同业产品互转；单位、同业产品允许互转。  
b) 产品大类（即产品模型，活期、定期等）之间不能互转；  
c) 转核算涉及的新旧科目状态不正常（已注销），拒绝处理；  
d) 转核算涉及的会计模型参数（科目组参数、产品科目对照表参数和产品码）未设置，拒绝处理；  
e) 当需要调整的账号与客户号不相匹配时拒绝处理;

4) 根据联机申请的记录，获取相关分户的账户余额信息（本金、表内计提利息）。  
5) 根据会计模型参数，获取分户相关的本金、表内计提利息、利息科目信息；

# 1.6.2.3 内部户与内部户控制表的关系

内部户控制表是对内部户属性的规范和控制，以保证内部户在业务核算过程中具有相同的财务意义和账务核算规则。内部户属性从控制表继承到账户层面，允许其具有特殊的属性（目前仅限挂账期限和借贷方限额）。

内部户控制表与内部户的控制关系如下：

1、修改内部户控制表时，内部户名称、允许发生方向、明细核算标识、联机更新标识、卡片核算标识、余额检查标识、应用标识允许修改，修改后同步内部户层相应属性。  
2、修改内部户控制表的内部户到期日，同时同步修改内部户层的内部户到期日、挂账到期日信息，相关内部户下的挂账到期日必须小于等于内部户到期日。  
3、内部户控制表记录状态为“待生效”，控制不允许新增内部户。  
4、内部户控制表记录状态为“停用”

1) “停用”状态的控制表不允许再新增内部户；  
2) 如果存量的内部户余额为 0 , 则注销该内部户;  
3) 如果存量的内部户余额不为 0 , 则内部户置为 “停用” 状态, 只允许往余额为 0 的方向发生交易, 直至余额为 0 , 则修改该内部户状态为 “注销”。

5、如果控制表的状态为“停用”，其相应的已销户的内部户不允许通过“激活”操作使其恢复正常；如果内部户控制表为“正常”状态，其相应的已注销的内部户允许通过“激活”操作使其恢复为“正常”状态。  
6、激活控制表不会自动激活其下的内部户，如果其下的内部户要求恢复正常，要手工激活。

# 1.6.3 挂销账处理原则

在银行，有一些业务处理不能即时完成，就需要挂账到某类内部户上，然后按相关要求在规定时限内完成的，进行销账处理，将挂在内部户上的资金做平，以保证业务的完整

性。挂销账是内部户基础上的挂销账，只要内部户属性设置了挂销账属性，通过该内部户发生的业务数据都会纳入挂销账机制。

# 1.6.3.1 挂销账业务规则

1、先挂账后销账，销账必须以挂账为前提，有销必有挂，销账必须输入挂账编号，有针对性地销账。挂账时，系统根据内部户的挂销账控制属性-挂销账记账方向，进行挂账处理，登记挂销账登记簿，对每一笔挂账，系统可事先设置预定到期期限，对规定期限内未核销的挂账，机构关门需检查挂账到期未销情况由主管授权方可关门，日终系统将产生相应挂账逾期提示报表。销账时，系统根据内部户的挂销账控制属性-挂销账记账方向，提示输入挂账编号进行销账处理，登记挂销账登记簿。  
2、挂销账种类分“一挂一销”、“一挂多销”、“多挂一销”、“多挂多销”。对于“一销”销账方式，销账金额应该和所有挂账金额一致才允许销账。对于“多挂”的挂账方式，如果挂账时输入挂账编号则该笔交易记在输入的挂账编号下并生成新的序号，如果不输入挂账编号，系统则自动生成新的一笔挂账编号；系统自动为每一笔挂账在挂账编号下生成挂账序号，到期日期以最新一笔的挂账到期日作为该挂账编号的到期日。如果一旦发起了销账处理或者挂账编号逾期，也不允许再进行挂账，只能生成一个新的挂账编号挂账处理。销账完成后，相应的挂账记录状态改为“结清”。  
3、日终批量时，系统会对设置了挂销账的内部户进行账挂核对，检查挂账登记簿中的挂账余额是否与相应内部户余额一致。检查挂销账总表金额、余额，与挂账明细、销账明细的核对检查。  
4、同级机构间不允许跨机构销账，上级机构可以为辖属机构销账。  
5、核心系统各业务子系统的挂销账，由业务子系统发起挂销账的业务处理，例如现金长短款的挂销账，由现金管理模块发起。会计信息子系统的挂销账管理只需要

负责挂销账的控制设置包括挂销账内部户的设置、期限的控制等，以及挂销账明细的登记和查询、机构关门的挂账到期情况检查、到期的报表提示。

6、系统提供通过手工综合记账系统进行手工挂销账处理，并同步登记更新挂销账登记簿。  
7、销账需要各相关系统在有关交易画面输入挂账编号触发核心系统进行销账处理。核心系统根据挂账编号进行销账的处理，处理成功后会计系统产生销账通知单，包含挂账编号、销账序号、销账金额、销账到期日的信息。  
8、挂账和销账均支持冲正处理。涉及到挂销账的内部户的冲正处理，由系统自动从原会计流水的日志中得到原挂账编号和销账序号，分别针对挂账冲正和销账冲正进行相应的处理。  
9、季末提供找零长短款自动销账处理，系统自动将找零长短款挂账金额结转至其他收入/其他支出。找零长短款挂销账科目可在[各应用挂账科目参数]设置。

# 1.6.3.2 挂销账冲账原则

# 1、挂账的冲账规则

1）已冲销的挂账记录，或者该挂账编号下挂账余额小于冲账金额，不能进行挂账的冲正处理。  
2) 如果挂账金额不等于冲账金额，不能进行挂账的冲正。  
3) 挂账的冲正处理成功，需要将挂账明细状态置为“已冲销”。  
4) 挂账冲正后，挂销账记录的挂账总金额需要扣减挂账冲账金额，重新计算挂账余额信息。如果挂账总金额不为0但挂账余额为零，挂销账记录状态置为“全部核销”；如果挂账总金额为0则将挂销账记录状态置为“已冲销”。

# 1.6.3.3 批量挂销账的业务规则

1、批量挂销账是为手工单笔记账提供批量处理的方式，并不是单独设立一个批量挂销账功能菜单。手工记账处理提供 EXECL 导入方式；  
2、批量挂账是逐笔产生挂账编号，并按照挂账编号逐笔登记挂销账登记簿。当批量销账时，可以根据挂账时输出的挂账编号，逐笔输入进行销账或者采用excel表批量导入销账。（由于批量挂账时，科目可能不一样、借贷方向可能不一致，所以只能逐笔产生挂账编号）。  
3、系统提供批量销账功能，将勾选的挂账记录未销金额逐笔全部核销到同一科目及顺序号的销账内部账号。

# 1.6.4.2 记账规则

# 1、表内科目

表内科目的记账采用借贷记账法，借贷记账法是根据复式记账原理，按照“有借必有贷，借贷必相等”的记账原则。表内科目的记账要求为按照账务机构、货币、起息日期借贷平衡。表内科目分为资产类、负债类、资产负债共同类、所有者权益、损益类。

# 2、表外科目

表外科目采用单式记账法，记录重要单证，如银行承兑汇票、贸易担保等。根据单式记账法，不要求机构、货币、起息日期借贷平衡。

# 1.6.4.3 账务处理原则

1、记账处理的总原则是实时更新客户分户、内部户，检查每笔交易整套会计分录的平衡性并生成会计传票。  
2、系统提供准实时的总账加工方式，提供总账数据的准实时查询功能；日终再根据当日汇总的交易传票进行再次加工，对准实时的总账数据进行核对检查。  
3、遵循交易核算分离原则，即核算出现异常不会影响交易的正常完成。核算异常采用自动通过核算异常科目垫平处理。

# 1.6.4.4 交易核算分离原则

交易核算分离是指交易发生时核算出现异常不会影响交易的正常完成。核算异常采用自动垫平方式处理，事后由用户人工跟进，并对垫平记录进行核销处理。

# 1、联机交易核算异常处理

1) 科目属性异常（非底层科目、科目状态“待生效”、“注销”），出提示警告信息，交易不拒绝，系统自动通过核算异常科目垫平。  
2) 内部户异常（余额方向不符、发生额方向不符、内部户状态“销户”），出提示警告信息，交易不拒绝，系统自动通过核算异常科目垫平。内部户停用状态，允许内部户往余额为0的方向进行交易。如果交易往内部户余额增大的方向处理，提示警告信息，交易不拒绝，系统自动通过核算异常科目垫平。  
3) 交易借贷不平，出提示警告信息，交易不拒绝，系统自动通过核算异常科目垫平  
4) 对于挂销账方式维持原处理方式。挂销账出现异常，交易报错并拒绝。  
5) 内部户余额超限，报错，交易拒绝。

# 2、手工记账交易核算异常处理

1) 科目属性异常（非底层科目、总账记录不存在且不允许自动生成总账、发

生额方向不符、科目状态“待生效”、“注销”），报错，交易拒绝。

2) 内部户异常（余额方向不符、发生额方向不符、内部户状态“销户”），报错，交易拒绝。内部户停用状态，允许内部户往余额为0的方向进行交易。如果交易往内部户余额增大的方向处理，交易拒绝。  
3) 交易借贷不平，报错，交易拒绝。  
4) 挂销账出现异常，报错，交易拒绝。  
5) 内部户余额超限，报错，交易拒绝。

# 3、日终批量交易核算异常处理

1) 科目属性异常（非底层科目、总账记录不存在且不允许自动生成总账、发生额方向不符、科目状态“待生效”、“注销”），交易不拒绝，系统自动通过核算异常科目垫平。  
2) 内部户异常（余额方向不符、发生额方向不符、内部户状态“销户”），出提示警告信息，交易不拒绝，系统自动通过核算异常科目垫平。内部户停用状态，允许内部户往余额为0的方向进行交易。如果交易往内部户余额增大的方向处理，提示警告信息，交易不拒绝，系统自动通过核算异常科目垫平。  
3) 交易借贷不平，出提示警告信息，交易不拒绝，系统自动通过核算异常科目垫平。

# 1.6.4.6 起息控制

# 1、起息日的限制：

1) 均通过参数控制能否发起倒起息和远期息交易。手工记账和应用系统产生账务的起息日的期限受开放周期的限制。  
2) 倒起息日期不得小于开户日期。

# 2、倒起息处理 (Back Value)

倒起息处理（Back Value）方式只更新当日总账和内部户的余额及发生额，对以前总账和内部户的历史余额都不产生影响，只会影响外部分户的历史余额。

# 1.6.4.7 冲账业务规则

1、年内错账都采用同向红字冲正，跨年采用反向蓝字冲正。重新补账（即倒起息交易）采用蓝字补账的原则。  
2、关于现金冲账，当天可以冲现金账，冲账计入现金科目。现金科目隔日冲账不能计入现金科目，要求只能通过现金冲账过渡科目（非现金科目），补账时将过渡科目冲销，采用挂销账机制，挂账类型为“一挂一销”。现金冲账过渡科目账户需设成“日终余额为  $0^{\prime \prime}$ ，机构关门不为零时需要主管授权。  
3、系统对冲账时限采用参数控制（开放周期参数）。

4、系统控制当日错账冲正必须是经办用户操作。无论当日错账或隔日错账的冲正必须主管授权。  
5、由于冲账涉及错账，在处理冲账及补账交易时应强制在系统录入重要要素：如错账日期，冲正原因，并由系统自动产生冲账传票。  
6、账务的冲正处理：

1) 利息冲正采用倒起息（Back Value）方式处理。  
2) 当日的冲正账务反映在当天。  
3) 当日发生冲正，原交易记录未打印存折或对账单，则交易冲正后冲正记录也不反映在存折或对账单上，否则需要将冲正记录反映在存折或对账单上。对于补账（倒起息）交易，倒起息记录按照交易日反映在存折或对账单上。  
4) 隔日的冲正，账务反映在当天，总账不会调整以往的总账余额，但是分户会自动调整之前的分户余额，例如存款会重新计算存款息和更新之前的本金。  
5) 对于冲账引起的计提差额部分、积数，由各应用系统进行调整。

7、在冲账交易完成后系统自动弹出相应的补账界面处理正确的账务，或授权无需处理。冲账后实时反映客户账（即内外部户账）。  
8、对于当日错账或隔日错账，都采用冲账的方式处理。对于当日冲账以及隔日冲账的交易均适用以下规则：对于外部户和内部户，发现的错账，账务引擎和核心交易系统能够进行追溯到原始的交易流水号，冲账后实时反映外部户和内部户。如果冲账时科目状态异常（注销、停用），则自动通过核算异常科目垫平。

9、涉及计提的错账，

1）涉及本结息周期的错账，系统从错账日开始至冲账日止重新计提，调整计提利息，账务均反映在冲账日当日。如涉及积数，系统自动计算正确积数，并作为冲账日的积数。  
2）涉及跨结息周期的错账，系统自动寻找冲账日前的最后一个结息日，计算从错账日起至最后一个结息日止的计提利息，账务反映在冲账日当日；再从最后一个结息日起至冲账日止重新计算积数和计提利息。跨结息日的调整利息，

与本结息周期的利息，最后在本结息日结息处理。

例如：活期账户每季20日结息，原错账交易日期为3月5日，冲账日为6月13日；

a) 冲账日前的最后一个结息日为3月20日；  
b) 计算错账日3月5日至3月20的调整利息；  
c) 根据错账金额重新计算3月20日至6月13日的积数，日终计算6月13日的计提利息；  
d) 本结息日期6月20日结息，结息金额包括b)的调整利息和3月20日至6月20日的计提利息

10、外围系统发送账务或交易给核心，其冲账操作由外围系统发起。  
11、如涉及迁移前的交易发生错账的，统一采用应急方案由信息科技部处理，以反向蓝字补账方式解决。  
12、挂销账冲正处理(参考“挂销账冲账原则”)。

# 1.6.5 手工记账原则

1、手工记账的功能是让业务部门在不能够通过应用系统业务交易生成的会计分录时的一种入账手段。  
2、手工记账功能控制规则如下：

1）手工记账只允许对内部户和纯科目进行手工记账，不允许对外部账户进行手工记账。  
2) 手工记账不允许对跨法人银行进行记账  
3) 手工记账不支持机构属性为分理处的核算机构。

4) 手工传票记账信息由柜面控制进行录入复核控制

3、手工传票记账规则：

1) 手工记账只允许录入内部户、纯科目（科目分户标识为“Y”、“G”）。  
2) 只有手工记账标识为“允许手工记账”的内部户和纯科目才能进行手工记账。  
3) 手工记账时的起息日受开放周期的限制。  
4) 手工记账的一套传票中，允许同时输入表内科目、表外的科目；且可以有不同的起息日期；对于表内科目要按不同的账套、账务机构、货币、起息日借贷平衡，若出现跨核算机构的传票，系统会自动将不同的核算机构通过行内往来科目轧差配平，以保持核算机构的账务平衡。对于表外科目不要求借贷平衡。

4、手工传票分录录入方式：

1) 手工传票可以通过手工传票联机交易手工录入。  
2) 手工传票可以通过模版的方式输入会计分录。  
3) 支持通过Excel建立传票并上传到系统。

# 1.6.6总账处理规则

1、系统提供准实时的总账加工方式，提供总账数据的准实时查询功能；日终再根据当日汇总的交易传票进行再次加工，对准实时的总账数据进行核对检查。  
2、总账账簿有机构、币种、科目、上期借贷方余额、当期借贷方发生额、期末借贷方余额。总账账簿涉及的科目为底层科目，统计时由底层科目逐级向上汇总至上级科目。涉及的机构为网点机构，可根据机构统计关系统计分行、总行总账信息。总账统计周期有日、旬、月、季、半年、年。

3、总账计算规则：

1）科目余额方向为借方：

本日借方余额  $=$  上日借方余额  $+$  本日借方发生额-本日贷方发生额

本日贷方余额  $= 0$

2）科目余额方向为贷方：

本日贷方余额  $=$  上日贷方余额  $+$  本日贷方发生额-本日借方发生额

本日借方余额  $= 0$

3）科目余额方向为双边反映：

本日借方余额  $=$  上日借方余额  $+$  本日借方发生额

本日贷方余额  $=$  上日贷方余额 + 本日贷方发生额

科目余额为正方反映：

如：本日借方余额>本日贷方余额则：

本日借方余额 = 本日借方余额 - 本日贷方余额

本日贷方余额  $= 0$

否则：

本日贷方余额 = 本日贷方余额 - 本日借方余额

本日借方余额  $= 0$

# 1.6.7 账务平衡检查规则

# 1.6.7.1 日常交易平衡检查

1) 手工记账

通过手工记账的分录，系统自动检查交易数据复式记账科目是否借贷平衡，如不平衡则交易失败。

2) 联机交易

系统自动检查交易数据复式记帐科目借贷是否平衡，如借贷不平衡，系统采用核算异常科目自动垫平机制处理。

3) 批次交易

批次处理交易处理，系统自动检查交易数据复式记账科目的借贷平衡性，如果出现借贷不平，系统采用核算异常科目自动垫平机制处理。

4) 外围发送会计传票平衡检查

实时检查外围发送会计传票的平衡性，由于外围交易系统未实现交易与核算分离的，所以核心检查交易数据不平衡则拒绝，返回错误信息给外围系统。

# 1.6.8 外围系统相关业务规则

# 1.6.8.1 外围系统的成账方式

系统支持外围系统的成账方式有两种：核心系统成账、外围系统成账。

1) 核心系统成账方式, 即外围系统将交易信息发送到核心系统, 通过核心的会计引擎拆分交易的会计传票信息。该方式遵循核心的交易核算分离机制, 即核算出现异常如科目状态异常或内部户异常, 交易不拒绝, 进行核算异常科目垫平处理, 联机可以通过总账准实时余额和总账历史交易明细查询核算异常科目垫平信息, 日终生成核算异常科目垫平清单。  
2) 外围系统成账方式，即外围系统（如国结系统）自行生成交易会计传票信息，传送到核心总账系统。核心需要对会计分录中的外部账户和内部户进行相关合法性检查和账务平衡检查。该模式不遵循交易核算分离原则，即核算出现异常如科目状态异常或内部户异常，交易拒绝，返回失败信息给外围系统。  
3) 核心实时检查外围送核心的交易信息唯一性，核心对外围送过来的交易日期及流水号进行唯一性检查，如出现重复则返回失败信息。

# 1.6.8.2 外围系统的对账机制

外围系统的对账方式主要是指交易明细核对

1) 交易明细核对：每日检查核心系统接收的交易请求处理结果是否与外围系统记录的一致，以便及时发现系统间错账。  
2) 核心可根据外围的联机对账请求提供交易处理成功或失败清单，也可以在日终批次处理生成交易处理成功或失败清单。  
3) 外围系统自行下载核心提供的交易处理成功或失败清单文件。  
4) 外围系统核对核心系统的交易清单与自身记录的是否一致，如果不一致则生成对账差错报表。

# 1.6.8.3 与大总账系统的关系

1、每日日终，核心将当日传票按照九个维度（机构段、责任中心段、会计科目段、参考段、产品段、条线段、内部往来段、来源段、备用段）统计借贷方发生额，发送给大总账系统。

# 1.6.12 分理处账务处理

分理处在机构属性上归属“1-营业机构”，分配独立的机构号，帐务上不独立核算，而是挂靠在其上级机构。

核心系统的传票、交易明细、合约及账户主档中记载的机构号全部是业务发生时最底层的机构号，也包括分理处机构号。

总账、内部账设置和加工时，如果涉及到分理处的账务，通过查找其上级机构，统一设置与加工到其上级机构，具体要求如下：

1、分理处不允许开立内部户，只允许在其上级机构设置内部户，系统控制内部户自动开户、手工开户时不允许在分理处层级。  
2、准实时总账加工、日终总账数据加工时，如果核算机构是分理处，则按照分理处所属上级机构（支行）来加工总账数据。总账数据包括支行层级、分行层级、总行层级的总账数据；  
3、合约及账户主档中记载的机构号全部是业务发生时最底层的机构号，总分核对、账卡核对时，按照将分户的最底层机构按照其上级机构汇总来进行总分核对和账卡核对。

# 1.6.13.1 计提规则

1、计提支持累加法和差额计提。  
2、计提周期为每日（按日计提）。

3、各种业务的具体计提规则，请参阅相关应用子系统功能书的详细描述。

# 1.6.13.2 开放周期参数

开放周期参数是总账系统参数中的基本属性之一，用来提供灵活的交易控制，系统设有两个开放周期（Open Period），分别定义倒起息交易与远起息交易的期限。

开放周期限制对各应用系统（同样也包括手工记账）产生的倒起息和远起息账务进行限制，如超出开放周期的范围，系统会拒纳该笔业务。

开放周期参数的维护由参数维护部门完成，通过公共子系统的机构总账参数维护交易实现。

开放周期参数的主要指标如下：

1. 倒起息开放周期：‘倒起息开放周期’是以月为单元的正数，可以为零。‘倒起息开放周期’为零时表示系统接受倒起息交易到本月月初。如果‘倒起息开放周期’为1时表示系统允许倒起息到上月月初，以此类推。  
2. 远起息开放周期：‘远起息开放周期’是以月为单元的正数，可以为零。‘远起息开放周期’为零时表示系统接受远起息交易到本月月末。如果“远起息开放周期”为1时表示系统接受到下月月末的远起息交易，以此类推。（海外用）

# 2.2 内部户管理

# 2.2.1 概述

内部户是银行根据本身业务经营管理需要而设置的内部专用账户，主要包括如下功能：

1、内部户控制表维护（包括控制表的新增、修改、停用、查询、激活、拷贝）  
2、内部户维护（包括内部户开户、修改、查询、销户、激活）；  
3、内部户交易查询（提供内部户历史交易明细查询）  
4、挂销账管理（包括挂销账信息查询、挂销账明细查询、批量销账处理、逾期未销提醒）  
5、手工记账交易（提供手工记账交易、手工记账 EXECL 上传处理、手工记账模板维护、手工记账传票综合查询）

6、定期批次处理（提供内部户控制表状态变更处理、内部户状态变更处理、内部户到期处理、内部户余额为0检查、账挂核对检查、账卡核对检查、或有资产或有负债检查）。
